4. Critical Engineering Problems & Solutions

실제 개발 과정에서 겪은 디버깅 여정입니다. 어떻게 발견하고, 어떤 가설을 세우고, 어떻게 해결했는지를 중심으로 작성했습니다.

Problem 1. "exit code -6" — 카메라 Race Condition

발견: SLAM과 YOLO를 동시에 ros2 launch로 실행하면, 간헐적으로 카메라 노드가 exit code -6으로 크래시했다. 5번 중 2~3번 발생하는 비결정적 오류라 재현이 까다로웠다.

조사: YOLO 단독 정상, SLAM 단독 정상. 동시 실행 시에만 발생. lsof /dev/video0으로 확인해보니, 두 프로세스가 거의 같은 시점에 카메라 디바이스를 열려고 시도하는 Race Condition이었다.

해결: 카메라 드라이버를 단일 프로세스로 분리하고, SLAM과 YOLO는 ROS2 토픽을 구독하는 구조로 재설계. camera_start.sh로 실행 순서를 제어했다:

# camera_start.sh — 핵심은 sleep 5
ros2 launch astra_camera astra_pro.launch.xml &
sleep 5  # USB 카메라 하드웨어 초기화 대기
ros2 launch gae_perception slam.launch.py

추가 해결: Ctrl+C 강제 종료 시 USB 좀비 프로세스 문제 → /etc/udev/rules.d/99-astra.rules에 영구 권한 규칙 작성으로 근본 해결.

Problem 2. SLAM + YOLO OOM 셧다운 — 8GB의 한계와 싸우기

발견: SLAM과 YOLO를 동시에 실행하면 Jetson이 5~10분 내에 OOM(Out of Memory)으로 셧다운. dmesg에 oom-killer 로그 확인.

원인: Jetson Orin Nano는 8GB RAM을 CPU/GPU가 공유(Unified Memory). 세 병목이 겹침: 1. SLAM 키프레임 무한 축적 (메모리 누수) 2. YOLO 매 프레임 추론 (GPU 독점) 3. 1080p 고해상도 처리 (특징점 추출 비용 과다)

해결 — 4단계 순차 최적화:

단계 조치 핵심 판단
1 SLAM Mem/Strategy=1 (Keep Last) 오래된 키프레임 자동 삭제
2 YOLO 5프레임당 1회 추론 0.17초 × 0.3m/s = 5cm, 장애물 감지 거리(0.5m) 대비 안전
3 TensorRT + FP16 변환 GPU 메모리 절반 — SLAM과 공존시키는 결정적 한 수
4 1080p → 320x240 (96% 감소) DetectionRate=1 + Grid/FromDepth=True로 맵 품질 유지

결과: SLAM + YOLO + 웹 스트리밍 동시 실행에서 RAM 안정. SLAM 1Hz 제한은 CPU 100%로 인한 모터 제어 지연 방지를 위한 우선순위 판단이었다.

Problem 3. HTTP Polling → MQTT — 아키텍처를 갈아엎은 이유

발견: 초기 Python Flask REST API + 1초 폴링 방식의 두 가지 문제: 1. 로봇 정지 중에도 매초 요청 → 불필요한 트래픽 2. 폴링 주기 1초 = 위치 정보 최대 지연 1초

Flask Bridge의 도전: ROS2 rclpy.spin() 이벤트 루프와 Flask HTTP 서버 루프를 하나의 프로세스에서 공존시켜야 했다. threading.Thread로 분리하고, TF2로 map → base_link 좌표 변환까지 구현. 동작은 했지만, HTTP 요청-응답 모델은 IoT 실시간 전송에 근본적으로 부적합했다.

MQTT 전환 효과: - 이벤트 기반 → 변화 시에만 발행 (네트워크 효율) - 발행 즉시 전달 → 지연 0.1초 이하 - Pub/Sub → 클라이언트 수에 무관한 확장성

배운 것: 단순히 라이브러리를 바꾸는 것이 아니라, 메시지 직렬화(JSON), 전송 빈도 제어(스로틀링), 연결 끊김 대응, QoS 레벨 선택 등 통신 프로토콜 수준의 설계가 필요했다.

회고 (3줄 요약)

  • 엣지 최적화: "최고 성능"이 아닌 "제한된 8GB 안에서 최선의 조합"을 찾는 트레이드오프 결정 과정이 가장 큰 성장이었다.
  • 아쉬운 점: INT8 양자화 미적용, SLAM 맵 자동 저장/복원 미구현, MQTT 자동 재연결 + 백오프 전략 미적용.
  • 다음 프로젝트: 통신 인터페이스(포트, 프로토콜, 토픽 형식)는 개발 초기에 합의하겠다.

5. Detailed Implementation

5-1. Visual SLAM (RTAB-Map) 구축 및 맵 제작

RTAB-Map 기반 Visual SLAM 시스템을 Jetson Orin Nano 환경에서 구축하고, 실내 자율 주행을 위한 2D/3D 맵을 생성했습니다.

주요 구현 내용:

  • RTAB-Map 파라미터 튜닝을 통한 Jetson 환경 최적화 DetectionRate=1 # 매 프레임 키프레임 검출 Mem/Strategy=1 # Keep Last 전략 (메모리 절약) Grid/FromDepth=True # 깊이 센서 기반 그리드맵 생성 Reg/Force3DoF=True # 3자유도 제한 (실내 평면 이동)
  • Orbbec Astra Pro RGB-D 카메라의 Color/Depth 이미지 동기화 (Approximate Sync, 0.05초 허용)
  • 카메라 → base_link 간 Static TF 퍼블리시 설정 ([0.1, 0, 0.1]m 오프셋)
  • 2D 점유 격자 맵 (map_2d.pgm, 5cm/pixel 해상도) 및 3D 포인트 클라우드 맵 (map_3d_cloud.ply) 생성
  • SLAM 위치 추정 결과를 MQTT로 브릿지하여 웹 프론트엔드에 실시간 로봇 위치 전송

흰 벽 환경 대응: 특징점이 부족한 흰 벽 복도에서 Odometry Lost가 빈발하여 Vis/MinInliers를 20→10으로 하향 조정. G2O 그래프 최적화가 이상치를 걸러주기 때문에 맵 품질 저하 없이 안정적 추적을 확보했다.

생성된 맵 파일: | 파일 | 형식 | 용도 | |------|------|------| | map_2d.pgm | 2D 점유 격자 이미지 | 내비게이션 경로 계획 | | map_2d.yaml | YAML 메타데이터 | 맵 해상도/원점 설정 | | map_3d_cloud.ply | 3D 포인트 클라우드 | 공간 시각화 | | rtabmap.db | RTAB-Map DB | SLAM 세션 저장/복원 |

5-2. MQTT 기반 VSLAM 좌표 ↔ 웹 실시간 통신

로봇(Jetson)과 웹 서버 간 VSLAM 좌표를 실시간으로 주고받기 위해 MQTT Pub/Sub 방식을 직접 설계하고 구현했습니다.

왜 MQTT를 선택했는가: - HTTP 폴링 대비 저지연, 저오버헤드 (초당 10회 위치 전송에 적합) - Pub/Sub 구조로 웹 클라이언트가 여러 대 접속해도 확장성 확보 - QoS 레벨 지원으로 메시지 전달 보장 가능

Pose Bridge 핵심 구현 (pose_bridge.py):

항목 구현 내용
ROS2 구독 토픽 /slam_pose → /rtabmap/localization_pose (remapping)
메시지 타입 PoseWithCovarianceStamped (공분산 포함 위치 추정)
MQTT 클라이언트 paho-mqtt v2 (CallbackAPIVersion.VERSION2)
브로커 주소 런치 파일 파라미터로 주입 (초기 1884 → 표준화 후 1883)
발행 토픽 robot/pose
페이로드 {"x": float, "y": float, "state": "active"} (JSON)

전송 최적화 기법:

# 1. 시간 기반 스로틀링 - 0.1초 쿨타임 (최대 10Hz)
if current_time - self.last_sent_time < 0.1:
    return

# 2. 거리 기반 필터링 - 1cm 이상 이동 시에만 전송 (v1)
dist = math.sqrt((x - self.last_x)**2 + (y - self.last_y)**2)
if dist < 0.01:
    return
  • v1 (초기): 시간 스로틀링(0.1초) + 거리 임계값(1cm) 이중 필터
  • v2 (개선): 거리 필터 제거, 시간 스로틀링만 유지 → 미세한 움직임도 웹에 반영

Deadband Filter 설계: 로봇 정지 시에도 SLAM 노이즈(~1mm)로 웹 마커가 떨리는 현상을 발견. 제어공학의 불감대 개념을 적용하여 2cm 미만 변화를 무시하는 필터를 설계했다. 이후 요구사항 변경에 따라 Deadband를 제거하고 스로틀링만 유지하는 방식으로 유연하게 전환.

통신 아키텍처 진화:

[Before] HTTP Polling (Flask, 1초 주기) → 지연 1초, 불필요한 트래픽
    ↓ MQTT Pub/Sub 전환
[과도기] 3-Port 파편화 (:1883, :1884, :9001) → 팀원마다 다른 포트
    ↓ 본인 코드 리팩토링 + 포트 표준화
[After] Dual Listener (:1883 TCP + :9001 WS) → 파라미터 주입 구조

5-3. YOLOv8 객체 인식 — TensorRT 최적화

YOLOv8 모델을 Jetson Orin Nano GPU에 최적화하여 실시간 객체 인식을 구현했습니다.

최적화 기법:

기법 적용 내용 효과
TensorRT 엔진 변환 .pt → .engine 변환 (레이어/커널 퓨전, 메모리 최적화) 추론 속도 대폭 향상
FP16 반정밀도 연산 half=True 설정으로 16비트 부동소수점 사용 메모리 2배 절감, 속도 2배 향상
프레임 스킵 5프레임당 1회 추론 (30FPS → 6FPS 추론) GPU 부하 80% 감소
입력 해상도 고정 640x640 고정 입력 (TensorRT 엔진 최적) 일관된 추론 성능 보장

추론 파이프라인:

카메라 RGB (320x240, 30FPS)
  → 프레임 스킵 필터 (5프레임당 1회)
  → YOLOv8 TensorRT 추론 (640x640, FP16)
  → 객체 검출 결과 + 깊이 센서 거리 계산
  → ROS2 토픽 발행 (/yolo_result, /yolo_image)

깊이 기반 거리 측정: 검출된 객체 중심의 10x10 픽셀 ROI에서 깊이값 중앙값 추출. 60cm 미만은 센서 한계로 0.5m 보정. 장애물 0.5m 이내 감지 시 자동 정지.
